# Output Kit — apply-yc Plugin

Formatting conventions for all output produced by apply-yc skills.

---

## Principles

- Compressed prose. No filler. Fragments OK.
- Every output section earns its place — if it doesn't change what the founder does next, cut it.
- Write in the founder's voice, not corporate voice. No buzzwords.

---

## Severity Indicators

Used in review and critique output:

| Indicator | Meaning |
|-----------|---------|
| `[KILL]` | This answer will likely kill the application if not fixed |
| `[WEAK]` | Passable but unconvincing — strengthen before submitting |
| `[OK]` | Serviceable — minor polish only |
| `[STRONG]` | Leave it |

---

## Section Labels

Use these labels consistently across skills:

```
**[Section: Question]**
```

Example:
```
**[Idea: Why is now the right time?]**
```

---

## Answer Blocks

Draft answers:

```
**[Section: Question]**
[Answer text]
```

Review output:

```
**[Section: Question]** — [KILL / WEAK / OK / STRONG]

Issues:
- [specific issue]

Strengthened:
[rewritten answer]
```

---

## Profile Blocks

Structured founder/company profile output:

```
## Company Profile
Name: [name]
One-liner: [one sentence]
Stage: [idea / prototype / revenue / growth]
[...]

## Founder Profile
[Name]: [role] — [relevant background in 2 sentences]
[...]

## Progress Snapshot
MRR: [$ or "pre-revenue"]
Users: [count or "0"]
Key milestone: [most impressive recent thing]
```

---

## Interview Simulation

```
**Q:** [question]
**A:** [answer]
**[OK / WEAK / KILL]** — [one line on why]
**Stronger:** [tightened version]
```

---

## Line Length

Keep answer drafts to the YC character/word limits where known. When unknown, default to:
- One-liner: 1 sentence, ≤15 words
- Short answer: ≤100 words
- Long answer: ≤300 words

---

## What to Avoid

- Bullet-point everything. Prose reads better on a YC form.
- Passive voice.
- "We are building a platform that enables..."
- Any sentence starting with "Leveraging..."
- Market size claims without a source or rationale.
